### Q1: CUDA Stream 的原理了解吗？

**我的回答**: CUDA stream 每个流按顺序提交 kernel。如果两个任务不需要等待前面的 kernel，可以放不同流实现并行；如果在同一个流里前面的 kernel 阻塞了，后面的都得等。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 方向对，但不够深入，缺少对 stream 并发执行模型的清晰描述。

**参考答案**:

CUDA Stream 是一个**按序执行的 GPU 命令队列**。核心要点：

1. **同一 stream 内**：kernel 严格按提交顺序执行（FIFO），前一个没完成后一个不会开始
2. **不同 stream 间**：可以**并发执行**，GPU 硬件调度器会把不同 stream 的 kernel 分配到不同的 SM 上
3. **默认 stream (stream 0)**：具有隐式同步语义，会等待其他 stream 完成

典型应用场景：
```
Stream 1: [H2D copy] → [Kernel A] → [D2H copy]
Stream 2: [H2D copy] → [Kernel B] → [D2H copy]
```
Stream 1 的 H2D 和 Stream 2 的 Kernel 可以同时执行（前提是硬件有空闲的 copy engine 和 SM）。

**关键补充**：
- CUDA 设备通常有 1 个 compute engine 和 1-2 个 copy engine (H2D / D2H)，所以 **数据传输和计算** 的 overlap 是最常见的优化
- `cudaStreamSynchronize()` 同步单个 stream，`cudaDeviceSynchronize()` 同步所有 stream
- Event 可以在 stream 之间建立依赖关系：`cudaEventRecord` + `cudaStreamWaitEvent`

### Q2: CPU 和 GPU 的 overlap 了解吗？

**我的回答**: CPU 提交到 model runner 之后可以做一些前处理后处理，GPU 去计算，CPU 做数据打包、前处理、后处理，实现 overlap。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 说了大概方向，但表述模糊，没有讲清机制。

**参考答案**:

CPU-GPU overlap 的核心在于 **CUDA 的异步执行模型**：

1. **CPU 提交 kernel 是异步的**：`kernel<<<...>>>()` 调用会立即返回，CPU 不会等 GPU 执行完
2. **CPU 继续干活**：提交完 kernel 后，CPU 可以做数据预处理、调度决策、准备下一批数据等
3. **GPU 在后台执行**：GPU 按 stream 里的顺序执行 kernel

在 vLLM 推理中的典型 overlap：
```
时间线:
CPU:  [准备 batch N+1 的输入] [调度决策] [采样/后处理 batch N]
GPU:  [================== 执行 batch N forward ==================]
```

具体在 vLLM 中：
- CPU 负责：tokenize、构造 attention metadata、调度 (scheduler)、采样 (sampling)、detokenize
- GPU 负责：model forward (embedding → attention → MLP → logits)
- **关键优化**：当 GPU 在跑当前 batch 的 forward 时，CPU 可以并行准备下一个 batch 的 metadata

### Q3: torch.compile 是干什么的？大概原理了解吗？

**我的回答**: 可以编译入图。底层不太了解，加那个指令后 PyTorch 可以入图，NPU 上还能进一步入图优化。

**自评**: <span class="score"><span class="score-dot score-bad"></span> 差</span> — 只知道怎么用，原理没讲清楚。

**参考答案**:

`torch.compile` 是 PyTorch 2.0 引入的**即时编译 (JIT) 框架**，核心流程：

```
Python 函数
    ↓ TorchDynamo (前端)
FX Graph (中间表示)
    ↓ TorchInductor (后端)
Triton kernel / C++ 代码
    ↓ 编译
可执行的优化代码
```

三个关键组件：

1. **TorchDynamo (前端捕获)**
   - 通过 Python 字节码级别的 frame evaluation hook 拦截执行
   - 将 PyTorch 操作捕获为 FX Graph
   - 遇到无法追踪的 Python 代码时做 "graph break"（切分成多个子图）

2. **AOTAutograd (自动微分)**
   - 在编译时就展开自动求导，生成前向+反向的联合图
   - 使得后端可以同时优化 forward 和 backward

3. **TorchInductor (后端编译)**
   - 默认后端，将 FX Graph 编译为 **Triton kernel** (GPU) 或 C++ (CPU)
   - 做 **算子融合 (operator fusion)**：把多个小算子合并为一个大 kernel，减少 kernel launch 开销和显存读写
   - 例如：`LayerNorm + Dropout + Add` 融合为单个 kernel

**为什么快**：
- 减少 Python 解释器开销（eager mode 每个 op 都要经过 Python dispatch）
- 算子融合减少显存带宽消耗
- 编译后的 kernel 经过了针对硬件的调优

```python
@torch.compile(mode="reduce-overhead")  # 减少 kernel launch 开销
def forward(x):
    return model(x)
```

### Q4: vLLM 里 model 文件怎么实现 TP？

**我的回答**: TP 对 num_head 进行除，词表也有切分，主要对权重切分。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 说了大方向但太简略，没有展开具体矩阵怎么切。

**参考答案**:

vLLM 的 model 文件通过两种 Parallel Linear 实现 TP：

**Attention 层**：
```python
# QKV 投影 — ColumnParallelLinear (按列切，按 head 分)
self.linear_qkv = ColumnParallelLinear(
    hidden_size,
    (num_heads + 2 * num_kv_heads) * head_dim,
    gather_output=False,  # 不 gather，每个 GPU 只持有部分 head
)
# Output 投影 — RowParallelLinear (按行切)
self.linear_proj = RowParallelLinear(
    num_heads * head_dim,
    hidden_size,
    input_is_parallel=True,  # 输入来自 ColumnParallel
)
# → AllReduce SUM
```

**MLP 层**：
```python
# gate_proj + up_proj — ColumnParallelLinear
self.linear_fc1 = ColumnParallelLinear(hidden_size, 2 * intermediate_size)
# down_proj — RowParallelLinear
self.linear_fc2 = RowParallelLinear(intermediate_size, hidden_size)
# → AllReduce SUM
```

**规范**：
- `ColumnParallelLinear`：权重沿 output 维度切分，**不通信**
- `RowParallelLinear`：权重沿 input 维度切分，**AllReduce SUM**
- 每层总共 **2 次 AllReduce**（Attention 一次 + MLP 一次）
- `num_heads % tp_size == 0` 且 `num_kv_heads % tp_size == 0` 必须满足
- 词表 Embedding 用 `VocabParallelEmbedding`，也按 TP 切分

### Q6: MLA 在公式层面怎么实现？(追问)

**我的回答**: 把 num_head 和 head_dim 两个维度变成一个固定的低维维度比如 512。

**自评**: <span class="score"><span class="score-dot score-bad"></span> 差</span> — 描述不准确，没有讲清 latent compression 的具体过程。

**参考答案**:

**MHA (Multi-Head Attention) 公式**：

$$Q = XW_Q, \quad K = XW_K, \quad V = XW_V$$

KV Cache 存储：每层存完整的 $K, V$，shape 为 $[seq\_len, n\_heads, head\_dim]$

**MLA (Multi-head Latent Attention) 公式** — DeepSeek-V2 提出：

核心：引入一个 **低维 latent 向量** $c_t$ 作为 KV 的压缩表示

$$c_t^{KV} = W_{DKV} \cdot x_t \quad \text{(下投影: } [h] \rightarrow [d_c] \text{, 其中 } d_c \ll n_h \times d_h \text{)}$$

$$K_t = W_{UK} \cdot c_t^{KV}, \quad V_t = W_{UV} \cdot c_t^{KV} \quad \text{(上投影: 推理时恢复)}$$

**KV Cache 只需存 $c_t^{KV}$**，维度从 $n_h \times d_h$（如 128×128 = 16384）压缩到 $d_c$（如 512）。

**关键点**：
- 训练时正常算 K, V，但 KV Cache 只存低维的 $c_t^{KV}$
- 推理时从 $c_t^{KV}$ 恢复 K, V（吸收到 attention 计算中，不需要显式恢复）
- KV Cache 压缩比：$d_c / (n_h \times d_h)$，通常 30x+ 压缩
- 同时对 Q 也做类似的 latent compression

```
MHA KV Cache: [seq_len, num_heads × head_dim]  → 很大
MLA KV Cache: [seq_len, d_c]                   → 很小 (d_c = 512)
```

### Q7: Linear Attention / Qwen3 系列了解吗？

**我的回答**: 不了解。

**自评**: <span class="score"><span class="score-dot score-bad"></span> 差</span>

**知识补充**:

**Linear Attention** 的核心思想是去掉 softmax，将 attention 复杂度从 $O(n^2)$ 降到 $O(n)$：

标准 Attention: $\text{Attn}(Q,K,V) = \text{softmax}(QK^T/\sqrt{d})V$

Linear Attention: $\text{Attn}(Q,K,V) = \phi(Q)(\phi(K)^T V)$

其中 $\phi$ 是特征映射。利用结合律先算 $\phi(K)^T V$（shape $[d, d]$），再乘 $\phi(Q)$，避免 $[n, n]$ 矩阵。

**优势**：推理时可以用 RNN 形式递推，每个 token 只需 O(1) 计算（相对序列长度）。

### Q9: vLLM 怎么管理请求状态？

**我的回答**: 有 waiting 和 running 两个队列，调度时看有没有显存分配给 waiting 的请求，有就放到 running，running 的请求去 model runner 跑。

**自评**: <span class="score"><span class="score-dot score-good"></span> 好</span> — 基本准确。

**参考答案**:

vLLM Scheduler 管理三个队列：

```
waiting → (有显存 & budget) → running → (显存不够) → swapped
                                ↑                        ↓
                                └──── (显存释放) ←────────┘
```

- **waiting**: 新到的请求，等待首次 prefill
- **running**: 正在 decode 的请求
- **swapped**: 显存不够时被 swap 出去的请求（KV Cache 暂存到 CPU）

调度逻辑（每一步）：
1. 检查 `running` 队列，是否有请求需要 preempt（抢占，因为显存不够）
2. 检查 `swapped` 队列，能否 swap in
3. 检查 `waiting` 队列，能否调度新请求 prefill
4. 受 `token_budget`（本轮最多处理多少 token）和 `max_num_seqs`（最多并发多少请求）约束

### Q10: vLLM 显存数据结构怎么设计的？(PagedAttention)

**我的回答**: 有 token budget 和 max request 两个维度控制。每个 request 有自己的 block table，映射到具体显存位置。decode 时根据 block table 取 KV Cache 数据，请求 finish 后释放 block table 对应的显存。

**自评**: <span class="score"><span class="score-dot score-good"></span> 好</span> — 理解了 PagedAttention 的核心设计。

**参考答案补充**:

```
┌─────────────────────────────────────────────────┐
│                  Block Manager                   │
│                                                  │
│  Free Block List: [blk_5, blk_8, blk_12, ...]  │
│                                                  │
│  Request A:                                      │
│    Block Table: [blk_0, blk_3, blk_7]           │
│    └→ 每个 block 存 block_size 个 token 的 KV   │
│                                                  │
│  Request B:                                      │
│    Block Table: [blk_1, blk_4]                   │
└─────────────────────────────────────────────────┘

物理显存 (KV Cache Pool):
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│blk_0 │blk_1 │ free │blk_3 │blk_4 │ free │ ...  │
│ Req A│ Req B│      │ Req A│ Req B│      │      │
└──────┴──────┴──────┴──────┴──────┴──────┴──────┘
```

关键设计：
- **Block**: 固定大小（如 16 tokens），KV 存储的最小单位
- **Block Table**: 每个请求一个，记录逻辑 block → 物理 block 的映射
- **Free list**: 空闲 block 池，新请求按需分配
- **Slot mapping**: 精确到 token 级别的位置索引，告诉 attention kernel 去哪里读写 KV
- 类似操作系统的 **分页机制**：逻辑连续，物理不连续，消除碎片
