### Q1: 单卡上 MoE 算子怎么优化？（不考虑通信，考虑 token 重排合并访存）

**我的回答**: 没答好，不太了解具体优化方法。

**自评**: <span class="score"><span class="score-dot score-bad"></span> 差</span> — 这个知识点完全没准备。

**参考答案**:

MoE (Mixture of Experts) 单卡优化的核心挑战：不同 token 被路由到不同 expert，导致**访存不规则、计算碎片化**。

#### MoE 前向计算流程

```
Input tokens: [T0, T1, T2, T3, T4, T5]
         ↓ Gate (Router)
Expert 0: [T0, T3, T5]    ← 不同 token 分到不同 expert
Expert 1: [T1, T4]
Expert 2: [T2]
         ↓ 各 Expert 独立计算 (FFN)
         ↓ 按原始顺序组合输出
```

**问题**：每个 expert 分配到的 token 数量不同（不均衡），且 token 在显存中不连续 → **访存效率差，GPU 利用率低**。

#### 优化一：Token 重排 (Permutation / Reorder)

**核心思想**：在进 Expert 前，把分配给同一个 Expert 的 token **物理上重排到一起**，计算完再恢复原始顺序。

```python
# Step 1: Gate 计算路由
gate_scores = router(hidden_states)          # [num_tokens, num_experts]
expert_ids = topk(gate_scores, k=top_k)      # 每个 token 选 top_k 个 expert

# Step 2: ★ Token 重排 — 按 expert 分组，连续存放
# 排序前: [T0(E0), T1(E1), T2(E2), T3(E0), T4(E1), T5(E0)]
# 排序后: [T0(E0), T3(E0), T5(E0), T1(E1), T4(E1), T2(E2)]
sorted_indices = argsort(expert_ids)
permuted_tokens = hidden_states[sorted_indices]  # 物理上连续

# Step 3: 计算每个 expert 的 token 数量和起始位置
expert_counts = bincount(expert_ids)    # [3, 2, 1]
expert_offsets = cumsum(expert_counts)  # [0, 3, 5]

# Step 4: 对每个 expert 调用 FFN (或 grouped GEMM)
for e in range(num_experts):
    start, end = expert_offsets[e], expert_offsets[e+1]
    output[start:end] = expert_ffn[e](permuted_tokens[start:end])

# Step 5: ★ Token 逆重排 — 恢复原始顺序
final_output[sorted_indices] = output
```

**重排带来的好处**：
1. **合并访存 (Coalesced Memory Access)**：同一 expert 的 token 在显存中连续 → GPU 可以一次性读取一整块，避免跳跃访存
2. **更大的 GEMM**：同一 expert 的 token 拼成一个大矩阵做一次 GEMM，比逐 token 或小 batch 的 GEMM 更能利用 Tensor Core

#### 优化二：Grouped GEMM / Fused MoE Kernel

与其 for 循环逐个 expert 调用 FFN，不如一次 kernel 处理所有 expert：

```
方案 A: Grouped GEMM (CUTLASS)
  - 一个 kernel launch 同时处理多个不同大小的矩阵乘
  - 避免多次 kernel launch 的开销
  - expert_counts = [3, 2, 1] → 一次 grouped GEMM 完成

方案 B: Fused MoE Kernel (如 Megablocks / vLLM 的实现)
  - 把 gate、permute、expert FFN、unpermute 融合进一个 kernel
  - 减少中间结果的 HBM 读写
```

#### 优化三：Padding to Aligned Size

由于每个 expert 分配到的 token 数不同，矩阵大小参差不齐。将每个 expert 的 token 数 **pad 到 Tensor Core 对齐的大小**（如 8 或 16 的倍数），虽然多算了一些，但 GEMM 效率大幅提升。

```
Expert 0: 3 tokens → pad 到 8  → GEMM [8, d] x [d, d']
Expert 1: 2 tokens → pad 到 8  → GEMM [8, d] x [d, d']
Expert 2: 1 token  → pad 到 8  → GEMM [8, d] x [d, d']
```

::: tip 面试要点
被问到 MoE 单卡优化时，答题框架：
1. **Token 重排** → 合并访存，减少不规则内存访问
2. **Grouped GEMM** → 一次 kernel 处理多个 expert，减少 launch 开销
3. **Padding 对齐** → 充分利用 Tensor Core
4. **Kernel 融合** → 减少中间结果的 HBM 读写
:::

### Q2: CUDA Graph 有啥用？

**我的回答**: 回答了基本概念，kernel launch 开销和图捕获。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 知道大概概念，但缺少具体场景和限制条件。

**参考答案**:

#### CUDA Graph 解决什么问题

每次 `kernel<<<...>>>()` 调用，CPU 端都要经历：参数校验 → 命令序列化 → 发送到 GPU driver → driver 调度。对于**大量小 kernel** 的场景，kernel launch overhead（通常 5-10μs/次）可能**超过 kernel 实际执行时间**。

```
无 CUDA Graph:
CPU: [launch K1] [launch K2] [launch K3] [launch K4] ...  ← 每次都有 overhead
GPU:         [K1]      [K2]      [K3]      [K4]
      ↑ gap ↑ gap ↑ gap ↑ gap                            ← kernel 间有空隙

有 CUDA Graph:
CPU: [launch Graph]                                        ← 一次提交
GPU: [K1][K2][K3][K4]                                      ← 无间隙连续执行
```

#### 使用方法

```python
# Step 1: 捕获
g = torch.cuda.CUDAGraph()
with torch.cuda.graph(g):
    # 在这里执行的所有 CUDA 操作会被录制，不会真正执行
    output = model(static_input)

# Step 2: 重放（多次调用，无 launch overhead）
static_input.copy_(new_data)   # 更新输入（必须是同一块显存）
g.replay()                      # 一次提交整个图
```

#### 关键限制

1. **输入/输出必须是固定大小的 tensor**：Graph 捕获时记录的是显存地址，replay 时地址不能变 → **不支持动态 shape**
2. **不能包含 CPU 控制流**：if/else、动态循环等都不行（因为 Graph 是静态的）
3. **不能包含显存分配/释放**：`torch.empty()` 不能在 graph 内部
4. **不能做 CPU-GPU 同步**：`torch.cuda.synchronize()` 不能在 graph 内

#### 在推理引擎中的应用

vLLM 在 **decode 阶段**大量使用 CUDA Graph：
- Decode 时每个请求只产生 1 个 token → kernel 很小、很多
- 把整个 decode forward pass 录成 CUDA Graph
- 按不同 batch size 预先捕获多个 Graph，运行时选最接近的
- Prefill 阶段由于 seq_len 变化大，通常**不使用** CUDA Graph

### Q3: NVIDIA MPS 是什么？以及 MPS 怎么控制 kernel 占用 SM 数量？

**我的回答**: 回答了 MPS 的基本概念和多进程共享 GPU。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 知道 MPS 基本概念，但不了解细粒度 SM 控制。

**参考答案**:

#### MPS (Multi-Process Service) 基础

正常情况下，多个进程使用同一个 GPU 时是**时间分片 (time-slicing)** 模式——同一时刻只有一个进程的 kernel 在执行，进程间通过 context switch 切换（开销大，~25μs）。

**MPS** 允许多个进程的 kernel **真正并发**在同一个 GPU 上执行，共享 GPU 的 SM 资源：

```
无 MPS (Time-Slicing):
GPU: [Process A kernel] → [ctx switch] → [Process B kernel] → [ctx switch] → ...

有 MPS:
GPU SM 0-39:  [Process A kernel]
GPU SM 40-79: [Process B kernel]    ← 真正同时执行
```

#### MPS 的 SM 分区控制

MPS 提供了 `CUDA_MPS_ACTIVE_THREAD_PERCENTAGE` 环境变量，可以**限制每个 client 进程最多使用的 SM 比例**：

```bash
# 限制该进程最多使用 50% 的 SM
export CUDA_MPS_ACTIVE_THREAD_PERCENTAGE=50
./my_process_A &

# 另一个进程也用 50%
export CUDA_MPS_ACTIVE_THREAD_PERCENTAGE=50
./my_process_B &
```

在 Volta+ 架构上，还可以用更细粒度的 API：
```c
// 设置进程级别的 SM 限制
cudaSetDevice(0);
// 通过 MPS control daemon 设置 active thread percentage
```

#### 关键特性
- **硬件隔离**（Volta+）：每个 MPS client 有独立的地址空间和故障隔离
- **优先级**：不同 client 可以设置不同优先级
- **资源限制**：可以限制每个 client 的显存上限 (`CUDA_MPS_PINNED_DEVICE_MEM_LIMIT`)

### Q4: 基于 MPS 的 SM 控制，怎么优化一个 stream 的 kernel 占用过多资源导致吞吐下降的问题？

**我的回答**: 有思路但答得不够系统。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span>

**参考答案**:

#### 问题场景

在推理引擎中，不同请求可能有不同的计算量（prefill vs decode）。如果一个 stream 上的 prefill kernel（计算密集）占用了所有 SM，导致其他 stream 上的 decode kernel（延迟敏感）被饿死：

```
问题：
Stream A (Prefill):  [████████ 大 kernel 占满所有 SM ████████]
Stream B (Decode):   [........等待 SM 空闲........] [kernel]  ← 延迟飙升
Stream C (Decode):   [........等待 SM 空闲........] [kernel]  ← 延迟飙升

整体吞吐反而下降，因为 decode 请求被阻塞、TTFT/TPOT 都变差。
```

#### 优化思路：用 MPS 做 SM 资源隔离

**方案：将 prefill 和 decode 拆到不同 MPS client，分配固定 SM 比例**

```
优化后：
MPS Client A (Prefill, 60% SM):  [████ prefill kernel ████]
MPS Client B (Decode, 40% SM):   [decode][decode][decode]  ← 不被抢占

SM 分配: |------- 60% Prefill -------|--- 40% Decode ---|
```

具体做法：
1. **Prefill worker** 和 **Decode worker** 跑在不同进程中
2. 通过 MPS 给 Prefill 分配 60% SM，Decode 分配 40% SM
3. Prefill 虽然变慢了一点（SM 少了），但 Decode 的延迟得到保障
4. **整体吞吐反而更高**：因为 decode 不再被阻塞，请求及时完成释放资源

#### 更进一步的优化

1. **动态调整 SM 比例**：根据当前 prefill/decode 的请求比例动态调整
2. **Chunked Prefill**：将长 prefill 拆分成多个小 chunk，穿插 decode，不完全依赖 MPS
3. **优先级调度**：MPS 支持 client 优先级，给 decode 更高优先级

::: tip 面试要点
这道题的考察核心是 **SM 资源的公平调度**。关键思路：
1. 识别问题：大 kernel 独占 SM → 小 kernel 饿死
2. 解决方案：MPS SM 分区隔离 or Chunked Prefill
3. 本质：在吞吐和延迟之间做 trade-off，通过资源隔离保证 SLO
:::

### Q5: Context Parallelism (CP) 和 Sequence Parallelism (SP) 了解吗？和 DCP (Decode Context Parallelism) 有啥区别？

**我的回答**: 不太了解，回答得比较含糊。

**自评**: <span class="score"><span class="score-dot score-bad"></span> 差</span> — 这块知识完全没有准备。

**参考答案**:

#### Sequence Parallelism (SP)

SP 通常和 Tensor Parallelism (TP) 配合使用。TP 把 Attention 和 FFN 的权重切分到多卡，但 **LayerNorm 和 Dropout** 等操作是对完整 hidden states 执行的，没被切分 → 每张卡上都有一份完整的 activation → 显存浪费。

**SP 解决方案**：在 TP 区域外的操作（LayerNorm, Dropout），沿 **sequence 维度** 切分：

```
Megatron-LM 的 TP + SP:

         TP 区域（Attention/FFN 权重切分到多卡）
         ┌─────────────────────────────────┐
Input → │ AllGather → Attn/FFN → ReduceScatter │ → LayerNorm(SP) → Dropout(SP)
         └─────────────────────────────────┘
         
SP 区域: 每卡只处理 seq_len/tp_size 长度的 token
TP 区域: 每卡处理完整 seq 但只有部分权重

通信: AllGather (SP→TP) 和 ReduceScatter (TP→SP) 替代了原来的 AllReduce
```

**本质**：SP 是 TP 的补充，把 TP 不切的 activation 也在 seq 维度上切分，减少 activation 显存。

#### Context Parallelism (CP)

CP 是专门为**超长序列**设计的并行策略。将输入序列沿 sequence 维度切分到多卡，每张卡只处理**一部分序列**的 Attention 计算：

```
Context Parallelism (CP=4):

Full sequence: [T0, T1, T2, T3, T4, T5, T6, T7, ..., T_N]
GPU 0: [T0, T1, ...]     处理 seq[0 : N/4]
GPU 1: [T2, T3, ...]     处理 seq[N/4 : N/2]
GPU 2: [T4, T5, ...]     处理 seq[N/2 : 3N/4]
GPU 3: [T6, T7, ...]     处理 seq[3N/4 : N]
```

**关键挑战**：Attention 需要每个 token 看到所有其他 token 的 K/V → 需要**跨卡通信 K/V**

实现方式（Ring Attention）：
1. 每张卡有本地的 Q、K、V
2. K/V 在 GPU 间做**环形传递**（类似 Ring AllReduce）
3. 每一步：用本地 Q 和当前收到的 K/V 计算 partial attention，累积结果
4. 经过 CP_size 步后，每张卡都见过了所有 K/V

```
Ring Attention (CP=4):
Step 0: GPU0 用 (Q0, K0V0), GPU1 用 (Q1, K1V1), ...
Step 1: K/V 环形传递 → GPU0 用 (Q0, K3V3), GPU1 用 (Q1, K0V0), ...
Step 2: K/V 环形传递 → GPU0 用 (Q0, K2V2), ...
Step 3: K/V 环形传递 → GPU0 用 (Q0, K1V1), ...
合并所有 partial attention → 最终结果 (Online Softmax)
```

#### SP vs CP 对比

| | Sequence Parallelism (SP) | Context Parallelism (CP) |
|---|---|---|
| **切什么** | Activation（LayerNorm/Dropout 处） | Attention 的 KV 序列 |
| **目的** | 减少 activation 显存占用 | 处理超长序列 (>128K) |
| **搭配** | 必须和 TP 一起用 | 独立的并行维度 |
| **通信** | AllGather + ReduceScatter | Ring KV 传递 |
| **适用** | 常规训练/推理 | 长上下文场景 |

#### Decode CP (DCP)

DCP 是 CP 在**推理 decode 阶段**的特化版本：

**Decode 阶段的特点**：每次只生成 1 个新 token（Q 长度=1），但要 attend to 整个 KV Cache。当 KV Cache 很长时，单卡放不下 → 需要 CP。

**DCP 与训练 CP 的区别**：
1. **Q 极短**（=1），KV 极长 → 通信模式不同：只需把 K/V 分片分布到多卡，Q 广播或复制
2. **无需 Ring Attention**：Q 很小，可以直接 AllGather KV 或 broadcast Q
3. **延迟敏感**：decode 是一个 token 一个 token 生成的，任何通信延迟直接影响 TPOT
4. **KV Cache 分布式存储**：每张卡只存 $1/\text{CP}$ 的 KV Cache，解决显存问题

```
DCP (Decode, CP=2):
GPU 0: 存 KV Cache 前半部分, Q(全部)
GPU 1: 存 KV Cache 后半部分, Q(全部)

每个 GPU 用完整 Q 和本地 KV 算 partial attention → ReduceScatter 合并结果
```

::: tip SP/CP/DCP 记忆口诀
- **SP** = TP 的显存补充，切 activation
- **CP** = 长序列的切分，环形传 KV（训练）
- **DCP** = CP 的 decode 特化，分布 KV Cache（推理）
:::

### Q6: 算法题 — GPU Kernel 调度与显存峰值

**题目描述**:

有多个 kernel 请求（不是同时到达），每个请求有：
- `start_time`: 最早可开始时间
- `end_time`: 截止完成时间（或持续时间）
- `mem`: 显存占用量

有 2 个 GPU，互相独立，每个 GPU **同一时刻只能执行一个 kernel**。
问：如何调度这些 kernel，以及两个 GPU 各自何时达到显存峰值？

**我的回答**: 说了动态规划的方向但没给出具体算法。

**自评**: <span class="score"><span class="score-dot score-ok"></span> 一般</span> — 方向对但没有完整思路。

**参考答案**:

#### 问题建模

```
Input:
  kernels = [(start_i, end_i, mem_i, duration_i), ...]  
  2 GPUs, each can run 1 kernel at a time
  
Goal:
  1. 合法调度：每个 kernel 在 [start_i, end_i] 内完成
  2. 找到最优调度策略（例如最小化 makespan 或最小化显存峰值）
  3. 计算每个 GPU 的显存峰值时刻
```

#### 解法思路：动态规划 / 贪心

**简化版**：每个 GPU 同一时刻只执行一个 kernel → 每个 GPU 的显存在任何时刻 = 当前正在执行的 kernel 的 mem（不是累加的），所以「显存峰值」= 该 GPU 上被分配的所有 kernel 中 mem 最大的那个执行时的那一刻。

**方法一：贪心调度 (Earliest Deadline First)**

```python
# 按 start_time 排序
kernels.sort(key=lambda k: k.start_time)

# 两个 GPU 各自的空闲时间
gpu_free = [0, 0]
gpu_peak_mem = [0, 0]
gpu_peak_time = [0, 0]
schedule = [[], []]

for kernel in kernels:
    # 选空闲较早的 GPU（如果都空闲选显存峰值更低的）
    # 确保 kernel.start_time >= gpu_free[g]
    g = 0 if gpu_free[0] <= gpu_free[1] else 1
    
    actual_start = max(gpu_free[g], kernel.start_time)
    actual_end = actual_start + kernel.duration
    
    gpu_free[g] = actual_end
    schedule[g].append((kernel, actual_start, actual_end))
    
    if kernel.mem > gpu_peak_mem[g]:
        gpu_peak_mem[g] = kernel.mem
        gpu_peak_time[g] = actual_start  # 峰值出现在该 kernel 执行时
```

**方法二：DP（当需要考虑截止时间约束时）**

将问题转化为 **二维背包/调度**：
- 状态：`dp[i][t0][t1]` = 前 i 个 kernel，GPU0 空闲时间为 t0，GPU1 空闲时间为 t1，是否可行
- 转移：第 i 个 kernel 放 GPU0 或 GPU1
- 优化：时间离散化，滚动数组

```python
# 离散化时间点
times = sorted(set(k.start for k in kernels) | set(k.end for k in kernels))
time_to_idx = {t: i for i, t in enumerate(times)}

# DP
# dp[t0][t1] = True 表示 GPU0 在 t0 时刻空闲、GPU1 在 t1 时刻空闲是可达的
dp = defaultdict(lambda: False)
dp[(0, 0)] = True

for kernel in sorted_kernels:
    new_dp = defaultdict(lambda: False)
    for (t0, t1), reachable in dp.items():
        if not reachable:
            continue
        # 放到 GPU 0
        if t0 <= kernel.start:
            finish = max(t0, kernel.start) + kernel.duration
            if finish <= kernel.deadline:
                new_dp[(finish, t1)] = True
        # 放到 GPU 1
        if t1 <= kernel.start:
            finish = max(t1, kernel.start) + kernel.duration
            if finish <= kernel.deadline:
                new_dp[(t0, finish)] = True
    dp = new_dp
```

#### 显存峰值分析

由于每个 GPU 同一时刻只跑一个 kernel，显存曲线是**阶梯形**（而非累积）：

```
GPU 0 显存:
     ┌───┐     ┌─────┐
     │ K1│     │ K3  │
─────┘   └─────┘     └─────── 
     ↑ peak = max(K1.mem, K3.mem)

GPU 1 显存:
  ┌─────┐   ┌───┐
  │ K2  │   │ K4│
──┘     └───┘   └─────────── 
  ↑ peak = max(K2.mem, K4.mem)
```

**每个 GPU 的显存峰值 = 分配到该 GPU 的所有 kernel 中，mem 最大的那个执行时。**

1. 先确认问题：kernel 的显存是否可以累加？GPU 同一时刻是否只能执行一个？
2. 给出简单贪心解法（EDF）
3. 讨论约束条件后升级到 DP
4. 分析显存峰值的计算方式
